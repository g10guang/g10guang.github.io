MySQL 中的类型

Apr 11, 2018

本文章采取的 MySQL 服务版本：mysqld Ver 5.7.20-0ubuntu0.16.04.1 for Linux on x86_64 ((Ubuntu))。</br> 以下讨论如无特殊声明则为 InnoDB 存储引擎。

MySQL 中有整形、浮点型、日期、字符串、文件、空间类型、JSON等，一行最长可以填充 65535 个字节，一个表中最多拥有 4096 列（Innodb 中是 1017）。

NDB 存储引擎中使用 4 字节对齐，假如一列的类型长度为 15 字节，由于对齐需要，会需要占用 16 字节空间。

顺带说一句 MySQL 虽然把默认存储引擎改为了 Innodb，但是我们心里面要清楚每个存储引擎都有其优势和劣势，其存在是解决什么问题，面相什么应用场景。MySQL 官方关于 NDB 与 Innodb 存储引擎的比较

tinyint(1) 与 tinyint(2) 的区别

对于整型这只是简单地改变类型的展示宽度，并不影响该类型的存储方式。

FLOAT(24) 与 FLOAT(64) 的区别

对于浮点型 M 不仅仅影响其展示宽度，还会影响其存储方式，因为不同精度需要不同的字节去存储。

DECIMAL / NUMERIC 使用二进制格式将 0~9 打包存储进 4 bits中，DECIMAL 中整数部分和小数部分分开存储。其存储锁需要的开销取决与左边数字 M：

DECIMAL 等固定长度的类型，DECIMAL(M, D) M 代表整个数字的精度，而 D 代表的是小数部分的精度。

  • M 对整形展示范围，对于浮点数和固定长度的类型 M 是类型可以存储的长度
  • D DECIMAL(M, D) D 最大只能是 30，D <= M-2
  • fsp 在 TIME DATETIME TIMESTAMP 中的应用
mysql> alter table inc change column salary salary DECIMAL(33, 31);
ERROR 1425 (42000): Too big scale 31 specified for column 'salary'. Maximum is 30.

mysql> alter table inc change column salary salary DECIMAL(100, 30);
ERROR 1426 (42000): Too-big precision 100 specified for 'salary'. Maximum is 65.

mysql> alter table inc change column salary salary DECIMAL(4,3);
ERROR 1264 (22003): Out of range value for column 'salary' at row 1

DECIMAL(4, 2) 存储 100.2 与 10.221 的差别。

  • 100.2 超出 DECIMAL(4, 2) 表达范围，无法存储，返回 out of range 错误
  • 10.221 只会保留精度为两位小数，所以相当于存储 10.22

点数的精度与本机的操作系统和硬件有关

日期格式的差别

Date Type Storage Required Before MySQL 5.6.4 Storage Required as of MySQL 5.6.4
YEAR 1 byte 1 byte
DATE 3 bytes 3 bytes
TIME 3 bytes 3 bytes + fractional seconds storage
DATETIME 8 bytes 5 bytes + factional seconds storage
TIMESTAMP 4 bytes 4 bytes + fractional seconds storage

DATETIME TIMESTAMP 支持秒后 6 位小数的精度

fractional seconds storage 是日期的秒小数部分，比如 DATETIME 中支持 ‘2018-04-12 00:00:00.999999’ 的时间格式，而根据不同的小数部分，需要不同大小的存储空间：

Frantional Seconds Precision Storage Required
0 0 bytes
1, 2 1 byte
3, 4 2 bytes
5, 6 3 bytes

可推测，秒的小数部分的存储方式类似 DECIMAL，将 0~9 的数字存储进 4 bits 中。

TIME(0), TIME(2), TIME(4) 和 TIME(6)分别使用 3 / 4 / 5 / 6 bytes，TIME 和 TIME(0) 等价，其中 TIME(M) 中 M 代表的是小数部分的位数。

YEAR 的存储是一个字节，能够表达的范围大小是 255，其能够表达的范围是 1901~2155。

DATE 只用于存储日期，没有时间部分（TIME），比如 ‘YYYY-MM-DD’，支持范围：’1000-01-01’ ~ ‘9999-12-31’。

DATETIME 既存储 DATE 也存储 TIME，展示的格式为：’YYYY-MM-DD HH:MM:SS’，支持范围：’1000-01-01 00:00:00’ ~ ‘9999-12-31 23:59:59’。

TIME 所使用的存储空间小于 TIMESTAMP 和 DATETIME，自然 TIME 能够表达的范围也就更小。

TIMESTAMP 既存储 DATE 也存储 TIME，展示的格式为：’YYYY-MM-DD HH:MM:SS’，支持范围：’1970-01-01 00:00:01’ UTC ~ ‘2038-01-19 03:14:07’ UTC，注意这里的 UTC 与时区无关。先将当地时间转化为 UTC 然后存储，读取时将 UTC 再转化为本地时间，对于国际化更加友好。但是如果改变了本地时区，读取出的时间与写入值不一样。

字符类型

Data Type Storage Required
CHAR(M) M*w bytes，0 <= M <= 255，w 是当前表中使用的编码方式所使用的最长字节
BINARY(M) M bytes，0 <= M <= 255
VARCHAR(M),VARBINARY(M) L+1 bytes 如果列所需要 0~255 bytes；否则 L+2 bytes
TINYBLOB, TINYTEXT L+1 bytes，其中 L < 2^8
BLOB, TEXT L+2 bytes，其中 L < 2^16
MEDIUMBLOB, MEDIUMTEXT L+3 bytes，其中 L < 2^24
LONGBLOB, LONGTEXT L+4 bytes，其中 L < 2^32
ENUM(‘v1’,’v2’,…) 1 or 2 bytes，取决于枚举中变量的数量（上限为 65535）
SET(‘v1’,’v2’,…) 1,2,3,4 or 8 bytes，取决于集和中元素的数量（上线为64）

上述表格中 L 是代表数据存储所需要的字节长度，因为数据长度不是固定的，所以需要记录每个数据的字节长度。

char(10) varchar(10) 指的是存储的字符数量，而不是字节数量，比如汉字在 utf-8 编码中为 3 字节，那么 char(10) 可以存储 10 个汉字，30 个字节。

当字符串长度超过能够类型所声明长度会产生什么后果？

char(10) varchar(10) 字符长度大于 10，则无法存储，Innodb 会报错。如果不在 strict sql 模式下，则会截断字符串，然后返回 warning。

varchar char 使用 GBK 和 utf-8 编码的不同

MySQL 支持非常多字符集，默认是 latin1，但是 latin1 不能够编码中文等亚洲字符，使得用更多的是 utf8，对移动端更加友好的 utf8mb4。utf8 一个中文字符的编码长度为 3 字节。10 个中文所需要的存储空间为 30 字节。

GBK 字符集对中文编码使用的是 2 字节。10 个中文所需要的存储空间为 20 字节。

char(M) 将会采用固定长度存储，CHAR 中存储如果字符长度不满 M，则会使用 Space ‘ ‘ 进行后缀填充，这样同样会带来一个问题，如何向 CHAR 字段中插入后缀需要带有空格的字符串呢？使用 varchar 吧。

MySQL 对字符串（CHAR / VARCHAR）进行比较时，不对后缀的空格进行比较：

mysql> select 'hello' = 'hello  ';
+---------------------+
| 'hello' = 'hello  ' |
+---------------------+
|                   1 |
+---------------------+

BLOB 与 TEXT 在 Innodb 中并不存储在聚集索引中，而是存储在外部文件中，在聚集索引中大于占据 9 到 12 个字节。

从 ENUM 和 SET 存储所需要的字节，可以看出其底层所使用的存储方式：

  • ENUM 将每一个枚举值映射到一个或者两个字节中，取值时候再采用一次映射
  • SET 字节中的每一位代表该值是否存在于当前集和中

MySQL5.7 对 JSON 的支持并不算很好，支持简单的 JSON 操作，但是还不支持对 JSON 进行索引，在这方面 MongoDB 可能表现得更加优秀。

InnoDB

数据存储在 Innodb 会被切分为页，数据会被存储在 Innodb 的聚集索引的叶子结点中。对于 VARCHAR 等可变常的字段，如果太长以至于在一页中存放不下时，Innodb 就会把该可变长的数据的部分存放在另一个磁盘页上，而多个磁盘页通过单链表进行连接，这行为叫页溢出。

顺带提一下，如果表中没有指定主键，Innodb 将会分配一个 6 字节的 ID，用于聚集索引。 Innodb 采用紧凑存储方式，使用 CPU 运算时间来换取存储和缓存命中率。
